# PromoFlow 统一搜索架构设计

## 1. 背景与目标

当前 PromoFlow 的 Web 搜索与 Bot 对话式搜索，底层都建立在同一条“查询文本生成 embedding → pgvector 相似度排序”的链路上。该实现具备基础的语义检索能力，但在实际业务体验上存在明显不足：

- 过度依赖单一路径的向量相似度，缺少传统关键词命中能力。
- 搜索 `视频`、`短视频`、`图片`、`图` 等类型词时，无法稳定转化为精准过滤条件。
- 搜索词命中素材标签（人工标签或 AI 关键词）时，结果排序缺少显著加权。
- 搜索 `玻璃瓶`、`食品` 等内容实体词时，无法充分利用标题、描述、标签、类目和 AI 摘要的综合信息。
- Web 搜索与 Bot 搜索虽然复用了同一个基础语义检索函数，但缺少统一的“查询理解、混合召回、融合排序、LLM 重排”能力。

本设计文档的目标是建设一套**统一的搜索服务体系**，同时服务于：

- Web 应用中的列表搜索、筛选与结果排序
- 飞书 Bot 的对话式素材检索与问答

并满足以下要求：

1. 同时兼顾**关键词精确命中**与**自然语言语义理解**。
2. 对类型词、标签词、类目词、实体词具备稳定且符合业务直觉的表现。
3. 支持轻量级 LLM 参与 **Query 理解** 与 **Top-K 重排**，但不依赖昂贵或过慢的大模型。
4. Web 与 Bot **必须共用同一套 search service 及其背后的检索/排序系统**，禁止出现两套独立搜索实现。

---

## 2. 设计原则

### 2.1 单一搜索内核

无论是 Web 页面搜索还是 Bot 对话式搜索，都应调用统一的搜索应用服务，统一使用：

- 相同的查询理解能力
- 相同的召回逻辑
- 相同的融合排序逻辑
- 相同的轻量 LLM 重排能力

区别仅体现在：

- Web：返回素材列表与解释性元信息
- Bot：在同一套检索结果基础上，进一步调用 LLM 生成对话回答

### 2.2 混合检索优先于纯向量检索

向量检索是必要能力，但不是唯一能力。最终系统应采用混合检索：

- 结构化过滤（content type / category 等）
- 关键词检索（标题 / 描述 / 标签 / AI 关键词 / 类目）
- 标签精确命中与实体词命中
- 向量召回（语义理解）
- 轻量 LLM 重排（只处理 Top-K）

### 2.3 业务直觉优先

对营销素材搜索而言，以下信号应被认为比抽象向量语义更强：

- 人工标签直接命中
- AI 关键词直接命中
- 标题直接命中
- 类型词触发的素材类型过滤
- 类目词直接命中

因此排序策略采用**方案 A：业务加权分数**，而不是完全依赖无解释的单一相似度分数。

### 2.4 轻量 LLM，快于“聪明过头”

LLM 在该场景中的定位：

- 做 Query 理解
- 做 Top-K 重排
- 做 Bot 最终回答生成

不要求使用高成本、高时延的大模型。优先选择响应更快、成本更低的轻量模型，例如当前项目已在使用的 `qwen3.5-flash` 级别模型。

## 3. 当前实现概览与主要问题

### 3.1 当前搜索流程

当前实现中：

1. 用户输入查询文本
2. 后端直接为该文本生成 embedding
3. 对 `contents.embedding` 执行 pgvector cosine similarity 排序
4. 返回排序结果

Bot 对话搜索的流程是在上述检索结果之上，再额外调用 LLM 生成自然语言回答。

### 3.2 当前实现的核心问题

#### 3.2.1 召回路径单一

当前主要依赖向量相似度，缺少：

- 关键词召回
- 标签精确命中召回
- 类目精确命中召回
- 类型词解析
- 业务排序加权

#### 3.2.2 结构化信息未充分参与检索

虽然内容实体中存在如下信息：

- 标题
- 描述
- 人工标签
- AI 关键词
- 一级/二级类目
- 内容类型（image / video）

但这些信息在当前搜索排序中并未被系统性使用。

#### 3.2.3 Web 与 Bot 只共享“最底层检索函数”，未共享完整搜索系统

当前共享的是基础 `semantic_search`，但未共享：

- 查询理解
- 多路召回
- 融合排序
- 重排策略

这会导致两个入口在后续演进中容易分叉，最终形成两套行为不一致的搜索体验。

#### 3.2.4 AI 元数据质量会直接影响向量检索上限

当前 embedding 主要基于 AI 生成的标题、摘要、关键词。若 AI 分析质量不稳定，或某些字段为空，则会显著影响向量质量。因此必须补强：

- embedding 文本构造策略
- 搜索文档构造策略
- AI 失败场景下的 fallback 处理

## 4. 目标能力

统一搜索系统应满足以下业务能力：

### 4.1 类型词理解

当用户输入以下词语时，应优先转化为结构化过滤条件：

- `视频` / `短视频` / `视频素材` → `content_type = video`
- `图片` / `图` / `海报` / `照片` → `content_type = image`

### 4.2 标签优先

当查询词直接命中：

- 人工标签
- AI 关键词

相关素材应在排序上得到显著提升。

### 4.3 实体词高可用

当用户搜索：

- `玻璃瓶`
- `食品`
- `礼盒`
- `胶原蛋白`

系统应综合利用：

- 标题
- 描述
- 标签
- AI 关键词
- AI 摘要
- 类目
- 向量语义

使真正包含该实体内容的素材尽量排在前列。

### 4.4 对话式检索复用同一内核

Bot 对话式搜索不应有独立召回系统。Bot 只是在统一搜索结果上追加：

- 对检索结果的解释
- 多条结果摘要整合
- 最终自然语言回答生成

---

## 5. 统一搜索总体架构

统一搜索应抽象为以下处理流水线：

1. **Query 预处理**
2. **Query 理解（规则 + 轻量 LLM）**
3. **多路召回**
4. **业务加权融合排序**
5. **Top-K 轻量 LLM 重排**
6. **统一结果输出**
7. **Bot 场景下可选的回答生成**

可概括为：

`用户输入 -> Query 解析 -> 多路召回 -> 业务加权排序 -> LLM 重排 -> 输出素材结果 -> (Bot 可选) 生成回答`

## 6. 服务边界与模块划分

### 6.1 统一服务入口

建议在 `backend/app/services/search/` 内建立统一搜索应用服务，例如：

- `query_parser.py`：查询预处理与 Query 理解
- `retriever.py`：多路召回
- `ranker.py`：业务加权融合排序
- `reranker.py`：Top-K LLM 重排
- `core.py`：统一编排入口
- `formatter.py`：Bot / Web 所需的结果整形辅助

统一对外暴露一个主入口，例如：

- `search_contents(...)`

并让以下调用方共享该入口：

- `backend/app/routers/search.py`
- `backend/app/bot/handlers.py`

### 6.2 Web 与 Bot 的职责分工

#### Web 搜索

调用统一 `search_contents(...)`，返回：

- 结果列表
- 分数
- 可选命中说明（后续可扩展）

#### Bot 搜索

先调用统一 `search_contents(...)`，拿到同一份候选结果，再：

- 拼接上下文摘要
- 交给轻量 LLM 生成对话式回答

即：

- **共享检索与排序**
- **分离展示与回答层**

这样既统一底层逻辑，也保证不同入口能输出适合自身交互形态的结果。

## 7. Query 理解设计

Query 理解分为两层：**规则解析优先，轻量 LLM 补充**。

### 7.1 规则解析

对高频、稳定、成本敏感的模式，优先使用规则和词典处理。

#### 7.1.1 类型词识别

维护一份类型词词典：

- `video`: `视频`、`短视频`、`视频素材`
- `image`: `图片`、`图`、`海报`、`照片`

若 Query 中出现明确类型词，则产出：

- `parsed_content_type`

#### 7.1.2 强关键词提取

对 Query 做基础分词与归一化，提取：

- 去噪后的关键词列表
- 保留原词顺序
- 去除明显停用词

#### 7.1.3 同义词归一

维护轻量同义词表，例如：

- `图` → `图片`
- `短视频` → `视频`
- `瓶装` ↔ `玻璃瓶`（谨慎处理，避免误伤）

同义词归一仅用于扩展召回，不应直接篡改用户原始查询语义。

### 7.2 轻量 LLM 理解

当 Query 满足以下条件之一时，可触发轻量 LLM：

- Query 较长
- 同时包含多个约束
- 出现模糊自然语言描述
- 规则解析结果置信度较低

LLM 输出结构建议包括：

- `normalized_query`: 归一化后的检索表达
- `content_type`: 识别出的素材类型（可为空）
- `must_terms`: 强约束词
- `should_terms`: 软约束词
- `intent`: 查询意图简述

示例：

用户输入：`找一些适合朋友圈宣传的玻璃瓶食品短视频`

期望解析：

- `content_type = video`
- `must_terms = [玻璃瓶, 食品]`
- `should_terms = [朋友圈, 宣传]`
- `intent = 查找食品类玻璃瓶包装的宣传短视频素材`

### 7.3 Query 理解输出结构

统一输出建议定义为内部领域对象，例如：

- `raw_query`
- `normalized_query`
- `parsed_content_type`
- `must_terms`
- `should_terms`
- `query_embedding_text`
- `need_llm_rerank`

## 8. 多路召回设计

多路召回至少包括四条路径。

### 8.1 路径一：结构化过滤召回

针对 Query 理解阶段识别出的结构化条件，优先约束候选集。

首期强制支持：

- `content_type`

后续可扩展：

- `category_id`
- `primary_category_id`
- 审核状态（目前默认仅 approved）

### 8.2 路径二：关键词 / 全文召回

基于 PostgreSQL 的全文检索（FTS）或等价方案，对如下字段建立统一搜索文档：

- 标题 `title`
- 描述 `description`
- 人工标签 `tags`
- AI 关键词 `ai_keywords`
- AI 摘要 `ai_summary`
- 一级类目名 `primary_category_name`
- 二级类目名 `category_name`
- 素材类型标签（图片 / 视频）

建议为搜索文档设置字段权重：

- A：标题、人工标签
- B：AI 关键词、类目名、类型标签
- C：描述、AI 摘要

### 8.3 路径三：标签精确命中召回

单独判断 Query 关键词是否命中：

- 人工标签完全相等
- AI 关键词完全相等
- 人工标签包含短语
- AI 关键词包含短语

该路径不是可有可无的细节，而是业务上非常强的信号，应参与显式加分。

### 8.4 路径四：向量召回

保留 pgvector 语义召回，用于处理：

- 自然语言表述
- 同义表达
- 未显式命中但语义相关的素材

但向量召回在统一系统中不再是唯一排序依据，而是候选结果的重要来源之一。

## 9. 搜索文档与 embedding 文本设计

### 9.1 搜索文档（供关键词检索使用）

每条素材应构造统一的“可检索文档”，逻辑上包含：

- 标题
- 描述
- 人工标签
- AI 关键词
- AI 摘要
- 一级类目
- 二级类目
- 素材类型中文标签

该搜索文档用于：

- 关键词检索
- 结果解释
- 可能的命中高亮

### 9.2 embedding 文本（供向量召回使用）

建议 embedding 文本不再仅由 `title + summary + keywords` 组成，而应扩展为：

- 标题
- 描述
- 人工标签
- AI 关键词
- AI 摘要
- 一级/二级类目
- 内容类型标签

这样可以显著提升向量对业务信息的覆盖度。

### 9.3 AI 失败 fallback

若 AI 分析失败，不应直接生成空文本 embedding。需要明确 fallback 规则，例如：

优先使用以下字段拼接非空文本：

- 用户输入标题（若有）
- 用户描述
- 人工标签
- 类目名
- 内容类型标签

只有在存在有效文本时才生成 embedding；否则标记 AI 失败，并允许后续重试或补算。

## 10. 融合排序设计（方案 A：业务加权分数）

本方案采用可解释、可调优的业务加权分数模型。

### 10.1 总体思路

对每条候选结果计算一个综合得分：

$$
score_{final} = score_{filter} + score_{exact} + score_{lexical} + score_{semantic} + score_{freshness}
$$

其中：

- `score_filter`：结构化条件匹配加分
- `score_exact`：精确命中类信号加分
- `score_lexical`：全文/关键词召回分数
- `score_semantic`：向量相似度得分
- `score_freshness`：轻量新鲜度加分（可选）

### 10.2 推荐初始分值

以下为推荐初始值，可在上线后通过日志与评估数据调优。

#### 结构化与精确命中

- 类型词解析后命中 `content_type`：`+80`
- 人工标签完全命中：`+100`
- 人工标签短语命中：`+85`
- AI 关键词完全命中：`+70`
- AI 关键词短语命中：`+55`
- 标题完全命中：`+60`
- 标题短语命中：`+45`
- 类目完全命中：`+40`
- 类目短语命中：`+30`
- 描述中出现 must term：每个 `+20`
- AI 摘要中出现 must term：每个 `+15`

#### 检索分值归一

- FTS 排名分数：归一到 `0 ~ 30`
- 向量相似度：归一到 `0 ~ 25`
- 新鲜度补充分：`0 ~ 5`

### 10.3 排序优先级解释

上述分值体现以下优先级：

1. **人工标签命中最高**
2. **AI 关键词命中次高**
3. **标题 / 类型 / 类目命中优先于纯向量相似度**
4. 向量分值作为补充，而不是压过结构化和精确命中的主排序依据

这能更符合营销素材检索中的业务直觉。

### 10.4 Must Terms 约束

若 Query 理解阶段产出 `must_terms`，则应对候选结果做额外约束：

- 若候选结果完全不包含任何 must term，可降权
- 若命中多个 must term，应显著加分

这对 `玻璃瓶`、`食品` 等明确实体词尤其重要。

## 11. Top-K 轻量 LLM 重排

### 11.1 何时使用

LLM 重排只处理前一阶段排序得到的 Top-K 候选，例如：

- Top 10
- Top 20

不应用于全库扫描，以避免高延迟与高成本。

### 11.2 LLM 重排的目标

LLM 重排用于解决以下问题：

- Query 是复杂自然语言表达
- 多个结果都命中关键词，但“真正符合意图”的排序仍不稳定
- 需要综合理解标题、摘要、标签、类目与类型后再决定优先级

### 11.3 轻量模型选择

建议使用当前已有的轻量模型，例如：

- `qwen3.5-flash` 或同等级别模型

理由：

- 响应快
- 成本低
- 足以胜任 Query 理解与候选重排

### 11.4 LLM 输入内容

对每个候选结果提供结构化摘要，例如：

- 标题
- 内容类型
- 人工标签
- AI 关键词
- 一级/二级类目
- 描述（截断）
- AI 摘要（截断）
- 业务加权初始分

让模型输出：

- 新排序顺序
- 可选的简短相关性说明

### 11.5 Web 与 Bot 的差异

- Web：可只使用重排后的结果顺序，不必展示复杂解释
- Bot：在相同重排结果上进一步生成自然语言回答

## 12. 统一搜索结果模型

统一搜索服务建议输出统一领域结果对象，包含：

- `content`
- `final_score`
- `lexical_score`
- `semantic_score`
- `matched_signals`
- `reranked`（是否经过 LLM 重排）

其中 `matched_signals` 可记录：

- 命中人工标签
- 命中 AI 关键词
- 命中标题
- 命中类目
- 命中类型词过滤
- 命中 must terms

该字段对：

- 后台调试
- 排序分析
- 未来搜索结果解释能力

都非常有价值。

## 13. 数据与索引设计建议

### 13.1 数据准备

建议为内容实体增加或派生以下检索辅助字段：

- `search_document`：统一全文搜索文本
- `search_tsvector`：PostgreSQL 全文索引字段（若采用持久化方案）
- `embedding_text`：用于生成向量的标准化文本（可选存储）

### 13.2 索引建议

首期建议建立：

1. `contents.embedding` 的 pgvector 索引
2. `title / description / ai_summary` 的全文检索索引
3. 标签关联表与类目字段的查询索引
4. `content_type`、`status` 的过滤索引

### 13.3 标签检索支持

由于标签分散在关系表与 JSONB 字段中，建议在检索路径上显式处理：

- 人工标签：通过关联表查询
- AI 关键词：通过 JSONB 或派生文本字段查询

必要时可将两者同步聚合进 `search_document` 以统一关键词检索入口。

## 14. API 与调用方式建议

### 14.1 Web 搜索 API

在现有 `/api/v1/search` 基础上扩展，但保持单一入口。

请求可逐步支持：

- `query`
- `limit`
- `content_type`（显式筛选，可由前端筛选器传入）
- `enable_rerank`（可选）

响应除素材外，后续可增加：

- `score`
- `matched_signals`

### 14.2 Bot 调用方式

Bot 不应自行实现检索逻辑，而应：

1. 调用统一 `search_contents(...)`
2. 取前 N 条结果
3. 生成对话回复

即保持：

- 一套召回排序系统
- 两类上层消费方

---

## 15. 实施路线建议

建议分阶段实施，避免一次性重构过大。

### Phase 1：修正基础数据质量

- 修复 AI 失败时的 fallback 策略
- 重构 embedding 文本构造逻辑
- 为存量数据补算 embedding / search document

### Phase 2：引入统一 Query 理解

- 增加规则词典
- 支持类型词解析
- 支持 must_terms / should_terms 提取
- 视情况接入轻量 LLM 做补充解析

### Phase 3：实现多路召回

- 新增关键词/全文召回
- 新增标签精确命中召回
- 保留向量召回
- 合并候选结果去重

### Phase 4：实现业务加权排序

- 落地方案 A 分数模型
- 输出匹配信号
- 对搜索结果进行可解释排序

### Phase 5：实现 Top-K 轻量 LLM 重排

- 仅对 Top-K 候选启用
- Web 与 Bot 共用重排逻辑
- Bot 在此基础上生成回答

### Phase 6：评估与迭代

- 对典型查询建立评测集
- 按查询类别分析排序效果
- 调整权重、词典和重排策略

## 16. 评估指标建议

为避免搜索体验只靠主观感受，建议建立简单评估集与观测指标。

### 16.1 离线评测样例分类

- 类型词查询：`视频`、`短视频`、`图片`
- 标签词查询：`胶原蛋白`、`女性营养`
- 类目词查询：`食品`、`保健品`
- 实体词查询：`玻璃瓶`、`礼盒`
- 复杂意图查询：`适合朋友圈宣传的食品类短视频`

### 16.2 指标

- Top 1 命中率
- Top 3 命中率
- Top 10 命中率
- 精确命中结果平均排名
- LLM 重排前后提升幅度
- 平均响应耗时（Web / Bot 分开观测）

## 17. 风险与注意事项

### 17.1 不能把 LLM 当主检索引擎

LLM 应用于理解与重排，而不是替代数据库检索。否则会导致：

- 时延过高
- 成本不可控
- 结果稳定性下降

### 17.2 不允许 Web 与 Bot 分叉演进

必须坚持：

- Web 搜索与 Bot 搜索共用同一套 search service
- 不允许分别维护两套召回和排序逻辑

否则长期会出现：

- 搜索结果不一致
- 维护成本翻倍
- 难以调优和评估

### 17.3 权重需要持续调优

业务加权分数是可解释的优势，也是需要调参的现实。首期权重应先上线，再通过真实查询日志逐步优化。

---

## 18. 结论

PromoFlow 的目标搜索系统不应是“纯向量相似度排序”，而应是一个**统一的、可解释的、混合式智能搜索系统**。

该系统的核心特征包括：

- Web 与 Bot 共用一套搜索内核
- 规则 + 轻量 LLM 做 Query 理解
- 关键词召回、标签命中、结构化过滤与向量召回并存
- 使用**方案 A：业务加权分数**做融合排序
- 使用轻量 LLM 对 Top-K 候选做重排
- Bot 在统一搜索结果基础上追加自然语言回答

这套方案既保留了语义检索的“智能”，也补足了关键词、标签、类目和类型过滤的“可控”与“精确”，更符合营销素材搜索场景对“好用”和“智能”的双重要求。
